### 2.2 模块依赖关系

```
app.py (主入口)
  ├── config.py      ← 常量配置（地区/行业枚举、默认采集源）
  ├── database.py    ← 数据库操作（CRUD、聚合查询）
  ├── crawler.py     ← 爬虫引擎（采集、解析、去重）
  │     └── cache.py ← 缓存操作（URL去重集合）
  └── cache.py       ← 缓存操作（查询结果缓存）
```

**关键依赖链**：
- `app.py` → `database.py` → `config.py`（数据库操作依赖配置常量）
- `app.py` → `crawler.py` → `cache.py` + `database.py`（采集链路依赖缓存和数据库）
- `app.py` → `cache.py`（直接使用缓存做查询结果缓存）

### 2.3 核心数据流

**采集数据流**：
```
用户输入关键词/URL
  → api_crawl() 接收请求
  → crawler.crawl_by_keyword() / crawler.crawl_by_url()
  → _generate_mock_data() 生成模拟数据
  → 对每条数据:
      → hashlib.md5(url).hexdigest() 计算 URL 哈希
      → cache.is_url_duplicate(hash) 检查 Redis 集合
      → url_exists(hash) 检查数据库
      → 均未命中 → insert_bidding() 入库 + cache.add_url() 加入集合
  → log_crawl_task() 记录任务
  → 返回 {success, count, data}
```

**查询数据流**：
```
用户设置筛选条件 → 点击查询
  → api_bidding() 构造 cache_key
  → cache.get(cache_key) 检查缓存
  → 命中 → 直接返回 {from_cache: true}
  → 未命中 → get_bidding_list() + get_bidding_count() 查询数据库
  → cache.set(cache_key, data, ttl=300) 写入缓存
  → 返回 {from_cache: false, data, total}
```

## 三、关键技术分析

### 3.1 去重机制分析

系统采用**三层去重**设计：

| 层级 | 位置 | 机制 | 代码位置 |
|------|------|------|---------|
| 第一层 | cache.py | Redis URL 集合（MD5 哈希） | `is_url_duplicate()` L69-L71 |
| 第二层 | database.py | 数据库 url_hash 唯一索引 | `url_exists()` L68-L74 |
| 第三层 | database.py | SQLite INSERT 唯一约束 | `insert_bidding()` L76-L103 (IntegrityError) |

**分析结论**：
- 第一层（内存/文件）快速拦截大部分重复，避免数据库查询
- 第二层（数据库查询）防止缓存丢失后的重复
- 第三层（唯一索引）作为最终防线，保证数据一致性
- **测试重点**：三层各自独立有效，且组合使用无漏洞

### 3.2 缓存策略分析

| 缓存类型 | 键格式 | TTL | 用途 |
|---------|-------|-----|------|
| 查询结果缓存 | `bidding:{region}:{industry}:...` | 300s (5分钟) | 列表查询结果 |
| 趋势数据缓存 | `trend:{days}` | 600s (10分钟) | 发布趋势 |
| 分布数据缓存 | `region_distribution` | 600s | 地区分布 |
| 分布数据缓存 | `industry_distribution` | 600s | 行业分布 |
| 分布数据缓存 | `keyword_analysis` | 600s | 关键词分析 |
| URL 去重集合 | `url_set`（MD5 哈希集合） | 永久 | URL 去重 |

**分析结论**：
- 缓存基于 JSON 文件实现，每次 set 都会全量写入磁盘，高并发下可能成为性能瓶颈
- TTL 比较机制使用字符串时间比较（`%Y-%m-%d %H:%M:%S`），精度为秒级
- **测试重点**：缓存命中/过期、缓存一致性（采集新数据后旧缓存是否影响展示）

### 3.3 前端架构分析

| 维度 | 实现方式 | 分析 |
|------|---------|------|
| 框架 | 原生 JavaScript（无框架） | 简单直接，但状态管理较原始 |
| HTTP 请求 | fetch API | 现代浏览器均支持 |
| 图表库 | ECharts 5.4.3 (CDN) | 功能强大，但依赖外网 CDN |
| 词云 | echarts-wordcloud 2.1.0 (CDN) | 有降级方案（analysis.js L152-L172） |
| 状态管理 | 全局变量（如 currentPage） | 简单场景足够，复杂交互需注意 |
| 样式 | 自定义 CSS + CSS 变量 | 响应式断点：1200px、768px |

**分析结论**：
- analysis.js 中热力图数据为前端随机生成（L181），与后端数据无关——这是一个**已知缺陷**
- 词云降级方案通过 try-catch 实现，逻辑合理
- list.js 中 perPage=15 与后端默认 per_page=20 不一致——**需在测试中验证**
- detail.html 中地区/行业显示的是 ID 而非中文名——**UI 体验问题**

## 四、风险与问题分析

### 4.1 已识别的代码问题

| 编号 | 问题 | 严重度 | 位置 | 说明 |
|------|------|-------|------|------|
| CODE-01 | 热力图数据造假 | 高 | analysis.js L181 | 前端随机生成数据，与数据库无关 |
| CODE-02 | list.js perPage 不一致 | 中 | list.js L2 vs app.py L124 | 前端 15 条/页 vs 后端默认 20 条/页 |
| CODE-03 | 详情页显示 ID | 中 | detail.html L38-L39 | region/industry 显示原始 ID 而非中文名 |
| CODE-04 | 密码 MD5 存储 | 中 | app.py L50, L54 | 安全性不足，建议 bcrypt |
| CODE-05 | cache.json 全量写入 | 低 | cache.py L25-L33 | 每次缓存操作都全量序列化写入文件 |
| CODE-06 | SQL LIKE 注入风险 | 低 | database.py L117 | `%keyword%` 未转义特殊字符 `%` 和 `_` |
| CODE-07 | keyword-analysis 分词粗糙 | 低 | database.py L214-L217 | 按空格分词标题，中文效果差 |
| CODE-08 | user_subscriptions 表未使用 | 低 | database.py L44-L52 | 建表但无任何代码引用 |

### 4.2 测试风险分析

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 爬虫为模拟数据 | 无法验证真实爬虫场景 | 标记为"模拟环境限制"，后续版本回归 |
| CDN 依赖（ECharts/词云） | 内网测试时图表不显示 | 验证降级方案；记录为环境前提条件 |
| JSON 文件缓存非线程安全 | 并发场景可能出现数据丢失 | 单用户测试环境下风险低，记录为已知限制 |
| SQLite 并发写入限制 | 高并发写入可能锁定 | 当前测试场景（手动操作）影响极小 |

## 七、结论与建议

### 7.1 总体评估

系统功能架构清晰，MVC 分层合理，核心功能（认证、采集、去重、筛选、分析）实现完整。代码质量整体良好，参数化查询防 SQL 注入、输入校验覆盖主要场景。主要风险点在于模拟数据环境和 CDN 外部依赖。

### 7.2 对测试的建议

1. **P0 用例必须 100% 通过**——涉及核心业务流程（登录→采集→筛选→分析）
2. **重点验证去重机制**——三层去重是系统核心亮点，需逐层验证
3. **缓存一致性**——采集新数据后，检查旧缓存是否被正确过期/更新
4. **记录 CODE-01~CODE-03 为缺陷**——热力图数据造假、perPage 不一致、详情页 ID 显示
5. **CDN 不可用时的降级验证**——断网或 CDN 不可达时，词云应降级为柱状图
